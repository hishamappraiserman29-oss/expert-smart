## 1. الغرض من هذه الوثيقة

تُلخّص هذه الوثيقة ما تم بناؤه في **وحدة التقييم الضريبي العقاري** (Tax Appraisal Module) كجزء من الإصدار الثاني (Phase 2) من منظومة Expert Smart.

تستهدف الوثيقة:
- مالك المنتج ومدير التقييم — للاطلاع على القدرات المُسلَّمة
- المسؤول التقني — لاستلام الوحدة والتحقق من اكتمالها
- المطوّر القادم — لفهم البنية والامتدادات المستقبلية
- العميل / صاحب المصلحة — للموافقة على النطاق والقيود

### ما الذي كان قبل Phase 2؟

كانت وحدة الضريبة تُجري حساباً أساسياً واحداً: القيمة الإيجارية × سعر الضريبة − الإعفاء = الضريبة السنوية، وتظهر النتيجة في بند واحد في الـ API. لم تكن هناك تفاصيل، ولا سياسات فئات، ولا تقرير Word مكتمل، ولا تكامل مع التقييم الجماعي.

### ما الذي وفّره Phase 2؟

بنى Phase 2 **سير عمل ضريبي متكامل**: من إدخال البيانات وتطبيق السياسة، مروراً بالحسابات الموسّعة والتقارير المتسقة، وصولاً إلى حزمة اعتراض استشارية — كل ذلك داخل المنصة الحالية دون تغيير البنية الأساسية.

## 2. ما الذي تم تسليمه

### أ. محرك الحساب الضريبي

التحسين الجوهري: توسيع الحساب من معادلة بسيطة إلى نموذج تقييم ضريبي كامل:

| المخرج | الوصف |
|--------|--------|
| `annual_rental_value` | القيمة الإيجارية السنوية = القيمة السوقية × نسبة الإيجار |
| `composite_factor` | المعامل المركب = معامل المحافظة × معامل البناء × معامل الموقع |
| `assessed_rental_value` | الوعاء الضريبي المعدَّل = القيمة الإيجارية × المعامل المركب |
| `exemption_amount` | قيمة الإعفاء المطبَّقة فعلياً |
| `taxable_value` | الوعاء الخاضع للضريبة بعد الإعفاء |
| `annual_tax` / `tax_due` | الضريبة السنوية المستحقة |
| `effective_tax_rate` | معدل الضريبة الفعلي من القيمة السوقية |
| `assessed_value` | القيمة المقدرة للأغراض الضريبية |
| `tax_sheet_summary` | ملخص ورقة الضريبة مضغوطاً لعرضه في التقارير |
| `appeal_narrative` | سرد نصي يشرح كيفية الوصول إلى النتيجة |
| `audit_notes` | قائمة ملاحظات التدقيق — تُسجّل الافتراضات المستخدمة |

**الثوابت الرياضية المضمونة:**
- `tax_due` يساوي دائماً `annual_tax`
- `taxable_value` يساوي دائماً `taxable_amount`
- القيمة الخاضعة لا تكون أبداً سالبة

### ب. المدخلات الضريبية الديناميكية

المقيّم يتحكم في كل متغير من متغيرات الحساب:

| الحقل | الوصف | الملء التلقائي؟ |
|-------|--------|----------------|
| `property_class` | فئة العقار — تحدد قيم السياسة الافتراضية | لا |
| `annual_rental_pct` | نسبة القيمة الإيجارية السنوية | **نعم** — من فئة العقار |
| `tax_rate` | سعر الضريبة | **نعم** — من فئة العقار |
| `exemption_threshold` | حد الإعفاء السنوي | **نعم** — من فئة العقار |
| `governorate_factor` | معامل المحافظة | لا — خاص بكل عقار |
| `construction_factor` | معامل تصنيف البناء | لا — خاص بكل عقار |
| `location_factor` | معامل الموقع | لا — خاص بكل عقار |
| `assessment_ratio` | نسبة التقدير | لا — خاص بكل عقار |
| `legal_basis` | السند القانوني | **نعم** — من فئة العقار |

### ج. سياسات الفئات العقارية (Policy Profiles)

**مُنفَّذة بالكامل.** النظام يعرف خمس فئات عقارية مُدرجة في الكود مع افتراضاتها:

| الفئة | نسبة الإيجار | سعر الضريبة | حد الإعفاء |
|-------|-------------|-------------|-----------|
| سكني (`residential`) | 3% | 10% | 24,000 ج.م |
| تجاري (`commercial`) | 8% | 10% | لا إعفاء |
| صناعي (`industrial`) | 6% | 10% | لا إعفاء |
| زراعي (`agricultural`) | 2% | 5% | 48,000 ج.م |
| إداري (`administrative`) | 8% | 10% | لا إعفاء |

عند اختيار المقيّم لفئة العقار، تُملأ الحقول الثلاثة تلقائياً في الواجهة. وإذا أدخل قيمة مختلفة، يُسجَّل التغيير كـ **تجاوز يدوي** (`manual_override`) مع ذكر القيمة الافتراضية والقيمة المستخدمة.

---

### د. حزمة الاعتراض الضريبي — `tax_appeal_package`

**مُنفَّذة بالكامل.** تُولَّد تلقائياً بعد كل تقييم ضريبي:

| المخرج | الوصف |
|--------|--------|
| `appeal_strength` | قوة الاعتراض: منخفضة / متوسطة / مرتفعة |
| `appeal_reasons` | أسباب مُرقَّمة برموز ودرجات حدة |
| `evidence_checklist` | قائمة بـ 6 مستندات مطلوبة أو مساندة للاعتراض |
| `operator_recommendation` | توصية المقيّم بناءً على التحليل |
| `appeal_summary` | ملخص مختصر لفرص الاعتراض |
| `formal_appeal_narrative` | مسودة أولى للسرد الرسمي |
| `disclaimer` | إخلاء مسؤولية واضح بالطابع الاستشاري |

> الحزمة **استشارية تحليلية** بالكامل. لا تُعدّ رأياً قانونياً ولا تمثيلاً أمام الجهات الضريبية.

### هـ. اتساق التقارير

**مُنفَّذ في ثلاثة مسارات:**

| المسار | الحالة | الملاحظة |
|--------|--------|----------|
| نافذة النتائج في الواجهة | ✅ مُنفَّذ | كل الحقول + حزمة الاعتراض بالألوان |
| تقرير Word | ✅ مُنفَّذ | قسم سياسة الفئة + حزمة الاعتراض كاملة |
| ورقة Excel — التقييم الجماعي (Mass Appraisal XLSX) | ✅ مُنفَّذ | 21 عموداً يشمل appeal_strength |
| ورقة Excel — العقار الفردي (xlsm) | ⚪ **غير مُنفَّذ** | القالب ثابت — Phase 3 target |

### و. تكامل التقييم الجماعي (Mass Appraisal)

**مُنفَّذ بالكامل.** صفوف `tax_assessment` في الدفعات الجماعية تحصل على:
- حسابات ضريبية كاملة لكل صف
- حقول السياسة وحزمة الاعتراض لكل صف
- ورقة مخصصة "Tax Assessment" في ملف XLSX التصدير

### ز. مقارنة السيناريوهات الضريبية — `tax_scenarios`

**⚪ غير مُنفَّذة — Phase 3 Target.**

الميزة مُخططة (Phase 2.5) لكن لم يُكتب لها كود حتى الآن. حقول `tax_scenarios` و`tax_sensitivity_summary` غائبة من استجابة الـ API في هذا الإصدار.

## 3. القيمة التجارية المُحقَّقة

| المشكلة السابقة | الحل المُسلَّم |
|----------------|---------------|
| تقارير Word تفتقد الحقول الضريبية المحسوبة | تقرير Word الآن يحصل على كل المخرجات الكاملة قبل الكتابة |
| تناقض بين الـ API والواجهة والتقرير | عقد موحد: نفس الحسابات تُغذّي الثلاثة |
| لا توثيق للافتراضات المستخدمة | `audit_notes` و`manual_overrides` يُسجّلان كل قرار |
| لا دعم للاعتراض الضريبي | حزمة اعتراض استشارية كاملة مع مستندات وسرد رسمي |
| معادلة ضريبية واحدة لكل أنواع العقارات | خمسة ملفات سياسة تعكس طبيعة كل فئة عقارية |
| التقييم الجماعي لا يدعم الضريبة | الدفعات الجماعية تُنتج حقولاً ضريبية كاملة لكل صف |
| صعوبة تسليم الملف بين الفرق | توثيق من ثلاثة ملفات + وثيقة تسليم تنفيذية هذه |

## 4. سير العمل الحالي

```
1. افتح المنصة على http://127.0.0.1:5000/
          │
2. اختر "الضريبة العقارية" كغرض التقييم
          │
3. أدخل بيانات العقار (الموقع / النوع / المساحة)
          │

          │

          │

   - ملف السياسة والتجاوزات اليدوية
   - حزمة الاعتراض إن وُجدت أسباب
          │
7. حمّل تقرير Word لتوثيق التقييم
   أو صدّر XLSX من وضع التقييم الجماعي
          │
8. استخدم المخرجات كدعم تحليلي — المراجعة القانونية النهائية
   مسؤولية المقيّم والمستشار الضريبي المرخّص
```

## 8. ملاحظات تشغيلية

### تشغيل المنظومة

```powershell
# بناء وتشغيل عبر Docker
docker compose -f deploy/docker-compose.yml up -d --build flask

# التحقق من حالة الخدمة
Invoke-RestMethod -Uri "http://127.0.0.1:5000/api/advisor/health" -Method Get
```

### الوصول للواجهة

افتح دائماً من: **`http://127.0.0.1:5000/`**  
لا تفتح ملف `frontend/index.html` مباشرة عبر `file:///` — طلبات الـ API ستُحجب.

### بعد أي تعديل على الكود

```powershell

docker compose -f deploy/docker-compose.yml up -d --build flask

# تحديث الواجهة في المتصفح
```

```powershell
python -c "import ast, pathlib; ast.parse(pathlib.Path('core_engine/bridge_api.py').read_text(encoding='utf-8')); print('AST OK')"
```

## 10. المرحلة التالية الموصى بها

### الخيار أ — Phase 3: محرك التقييم الجماعي المتقدم (الموصى به)

**التوصية:** الانتقال إلى Phase 3 — Advanced Mass Appraisal Engine.

**المبرر:**  
وحدة الضريبة الآن مستقرة ومُختبرة ومُوثَّقة. الأساس المنفرد والجماعي كلاهما قوي. الخطوة المنطقية التالية هي تعميق قدرات التقييم الجماعي نحو:
- مناطق السوق وتجميع العقارات بالموقع
- بوابات جودة البيانات والإشعارات التلقائية
- معايرة النماذج (Calibration) ودراسات نسب التقدير (Ratio Studies)
- حوكمة الدفعات: batch_id، سجل التدقيق، موافقة المراجع

**الوقت المقدر:** 6-8 أسابيع للإصدار الأول من Phase 3.

---

### الخيار ب — Phase 2.10: حزمة التصدير الرسمي للجهات التنظيمية

تُضاف قوالب تصدير بتنسيقات مصلحة الضرائب / هيئة الرقابة المصرفية، مع إمكانية رفع ملفات التقييم رقمياً.

**مناسب إذا:** كان هناك متطلب تنظيمي فوري أو عقد حكومي يستلزم تنسيقاً رسمياً.

**التوصية:** تأجيله لما بعد Phase 3 ما لم يكن إلزامياً.

## 12. ملاحظة ختامية

وحدة التقييم الضريبي المُسلَّمة تُوفّر **دعماً تحليلياً متكاملاً** لاحتساب الضريبة العقارية وتقييم فرص الاعتراض، مع تسجيل دقيق للافتراضات والسياسات المستخدمة في كل تقييم.

غير أن **التفسير القانوني النهائي وتقديم الإقرارات الضريبية الرسمية** يبقيان من مسؤولية مستشار ضريبي مرخّص ومحامٍ متخصص. النظام أداة تحليل ودعم قرار — لا أداة تقديم رسمية.
